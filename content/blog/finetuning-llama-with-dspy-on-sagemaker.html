I ranked 47 of 198 in the AWS AI League fine-tuning track, training Llama 3.2 3B with LoRA on SageMaker. The training data was fully synthetic, generated through a two-stage DSPy GEPA pipeline. Code is in tosinamuda/llama-finetuning-dspy.

Three things from this build are worth writing down: the pipeline shape, the JSONL choice, and the abstraction I wish I had named upfront.

The task

Generate one-page executive slides and consulting summaries on demand. The fine-tune target was a small open model that could run cheaply and respond in a consistent format. The training set had to be diverse enough to cover realistic business scenarios without me hand-writing thousands of examples.

The pipeline

Two GEPA-optimized prompts chained together, fed by a structured input grid.

Inputs are six variables that describe a scenario:

  • client_size
  • industry
  • pain_point
  • requirement
  • existing_stack
  • constraint

Stage A turns those variables into an instruction. Stage B turns the instruction into a response. Both prompts are optimized with GEPA against small hand-authored gold datasets:

  • instruction_dataset.jsonl — variables paired with a gold expected instruction. Used to optimize Prompt A.
  • response_dataset.jsonl — instruction paired with a gold expected response. Used to optimize Prompt B.

Generation runs the two optimized prompts over input_data.jsonl, a Cartesian grid of 1,098 variable combinations. The output is generated_output.jsonl, which is the actual training set that goes to SageMaker:

{ "instruction": "...", "context": "", "response": "...", "category": "brainstorming" }

context stays empty by design — every fact lives in instruction. Keeps the schema flat and matches what the SageMaker training job expects.

Why JSONL felt weird, and why it is right

Coming from web work, my reflex was JSON: one file, one array, parse it whole. JSONL — one JSON object per line, no outer array, no commas between records — felt wrong at first. Why throw away the structure?

The reasons stacked up the further I went:

  • Streaming. You can read, write, and append one record at a time without parsing the rest. A 1,098-row generation job that crashes at row 700 still has 700 valid records on disk.
  • Concurrency. Multiple workers can append safely. With a JSON array, every writer has to know about every other writer.
  • Diff-friendly. Adding a row is a one-line diff. Adding a row to a JSON array changes the closing bracket and (depending on style) the previous line's trailing comma.
  • Tooling. SageMaker, OpenAI fine-tuning, BigQuery, Spark, and most ML data pipelines treat JSONL as the default. The format is the contract.
  • Crash recovery. A truncated file is still parseable up to the last newline. A truncated JSON array is unparseable, full stop.

JSONL is JSON optimized for the ways data pipelines actually fail and grow. Once that clicked, I stopped reaching for arrays.

The abstraction I should have named on day one

I shipped the pipeline before I noticed the right shape. The codebase talks about "instruction" and "response" as if they were special, but they are not. They are two instances of the same thing: a Stage.

A Stage is:

  • an input schema
  • an output schema
  • a small set of gold training examples (input → output)
  • a GEPA-optimized prompt

You chain stages by feeding stage n's output into stage n+1's input. Two stages today, but the same shape would extend to three or five without rewriting the orchestration.

Naming it would also have forced a clearer split between the two roles JSONL plays in this repo, which I currently conflate:

  • Training examples — gold input/output pairs that GEPA uses to optimize a prompt. Hand-authored. Irreplaceable. (instruction_dataset.jsonl, response_dataset.jsonl.)
  • Generation grid — variable rows fed through optimized prompts at production time. Cartesian. Regenerable from a schema. (input_data.jsonl.)

Only the training examples are precious. The grid is enumeration. The optimized prompts and the generated output are fully reproducible from (training examples + grid + code), so they belong in build artifacts, not source.

What I would change

  • Refactor around Stage as the core type. Each stage owns its schemas, its gold set, and its optimized prompt directory.
  • Move input_data.jsonl generation behind a script that takes the variable schema and emits the grid. The committed file is just a checkpoint.
  • Drop the abandoned LLM-based seed expansion (seed.py). It was a side experiment, not part of the canonical flow.
  • Treat optimized_*_program/ and generated_output.jsonl as build outputs. Ignore them in git.

The competition rank was a useful forcing function. The architecture lessons came from looking at the repo a week later with no deadline pressure.
