Six bugs and antipatterns I keep finding in React code — mine and other people's. Each one has a clear failure mode and a clear fix.

1. Referential stability

Every render creates fresh object and function literals. Anything that compares by reference — useEffect dependencies, memo equality, prop diffing in child components — sees a "change" on every render even when the data is identical.

Antipattern:

function Parent() {
  const config = { limit: 10 };           // new object every render
  const onSelect = (id) => load(id);     // new function every render
  return <Child config={config} onSelect={onSelect} />;
}

function Child({ config, onSelect }) {
  useEffect(() => {
    fetchData(config);                    // re-runs every render
  }, [config]);
}

Fix: hoist stable values out of the component, memoize, or pass primitive deps instead of containers.

const CONFIG = { limit: 10 };           // module scope, stable

function Parent() {
  const onSelect = useCallback((id) => load(id), []);
  return <Child config={CONFIG} onSelect={onSelect} />;
}

function Child({ config, onSelect }) {
  useEffect(() => {
    fetchData(config);
  }, [config.limit]);                   // depend on the primitive
}

Rule of thumb: if a value goes into a dependency array, it has to be either a primitive or referentially stable.

2. Event listeners that never unmount

Adding a window or document listener in useEffect without returning a cleanup. Every re-run stacks another listener, and the component can update state after unmount.

Antipattern:

useEffect(() => {
  window.addEventListener('resize', onResize);
}, []);

Fix: always return the cleanup. Pass the same function reference to remove that you passed to add.

useEffect(() => {
  window.addEventListener('resize', onResize);
  return () => window.removeEventListener('resize', onResize);
}, [onResize]);

If onResize is not stable, wrap it in useCallback or store the latest version in a ref so the listener identity is stable across renders.

3. setTimeout vs requestAnimationFrame

setTimeout fires on the macrotask queue. It is not aligned with the paint cycle, so visual updates driven by it can run twice before a paint or skip a frame entirely. The result is jank.

For anything visual — animations, scroll-driven UI, layout-dependent measurement — use requestAnimationFrame. The browser calls it once per paint, right before the next frame, with the right timing.

Antipattern:

setTimeout(() => setX(x + 1), 16);     // hopes for 60fps

Fix:

useEffect(() => {
  let raf;
  const tick = () => {
    setX((x) => x + 1);
    raf = requestAnimationFrame(tick);
  };
  raf = requestAnimationFrame(tick);
  return () => cancelAnimationFrame(raf);
}, []);

setTimeout is the right tool for non-visual delays: debouncing input, retry backoff, polling. Not animation.

4. Timeouts that never clear

The same shape as the listener bug. A setTimeout in useEffect without a cleanup keeps firing after the component unmounts. The callback closes over stale state and may call setState on a dead component.

Antipattern:

useEffect(() => {
  setTimeout(() => setReady(true), 1000);
}, []);

Fix:

useEffect(() => {
  const id = setTimeout(() => setReady(true), 1000);
  return () => clearTimeout(id);
}, []);

Same goes for setInterval (clearInterval), requestAnimationFrame (cancelAnimationFrame), fetch (AbortController), and any subscription you create.

5. Derived state in useEffect

Using useEffect to copy or transform props into state. The component renders once with stale state, the effect runs, setState queues another render. Two renders, possible flicker, two sources of truth.

Antipattern:

function FullName({ first, last }) {
  const [full, setFull] = useState('');
  useEffect(() => {
    setFull(`${first} ${last}`);
  }, [first, last]);
  return <span>{full}</span>;
}

Fix: compute it during render. Use useMemo only if the computation is genuinely expensive.

function FullName({ first, last }) {
  const full = `${first} ${last}`;
  return <span>{full}</span>;
}

If the value depends on props and something the user changes, lift the user's input into state and derive the rest. Reach for useEffect only when you actually need to synchronize with something outside React (network, DOM, subscription).

6. Callbacks that depend on other callbacks

A useCallback whose dependency array includes another useCallback. Any input change to the inner callback invalidates the outer one, which invalidates anything that depends on it. Memoization unravels and you get the same re-renders you were trying to prevent.

Antipattern:

const formatItem = useCallback((item) => `${item.name} (${currency})`, [currency]);

const handleSelect = useCallback((item) => {
  log('selected', formatItem(item));
}, [formatItem]);                       // chained dep

Fix: store the latest function in a ref so the outer callback can call it without depending on it.

const formatItemRef = useRef(formatItem);
useEffect(() => { formatItemRef.current = formatItem; });

const handleSelect = useCallback((item) => {
  log('selected', formatItemRef.current(item));
}, []);                                 // stable

If you find yourself building long chains of useCallback, that is usually the signal to merge them into a single callback or to move the work into a reducer or external store.

The pattern behind the patterns

Five of these six bugs come from the same root: React re-runs your function on every render, and anything you create inside that function is fresh each time. The fixes are all variations of the same move — make the value stable (module scope, ref, memo) or make the dependency a primitive instead of a container.

The sixth, setTimeout vs requestAnimationFrame, is not a React bug at all. It just shows up in React code most often because that is where most UI work lives now.
